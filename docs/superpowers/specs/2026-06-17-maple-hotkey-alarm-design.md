# Maple Hotkey Alarm Design

## Goal

Build a Windows desktop alarm app for MapleStory farming sessions. The app should feel like a small web UI, but it must keep working while MapleStory is focused, while the app window is minimized, and while the app is hidden in the tray.

## Platform Choice

Use Electron.

The renderer process provides the web-style UI. The main process owns native behavior: global hotkey registration, tray behavior, desktop notifications, timer lifecycle, local persistence, and app packaging.

Pure browser/PWA delivery is out of scope because it cannot reliably detect hotkeys while another app is focused and cannot keep running after the browser is closed.

## User Experience

The first screen is the usable timer manager, not a landing page.

The app shows a list of alarm slots. Each slot contains:

- Name
- Hotkey
- Duration
- Remaining time
- Running/idle/expired state
- Register status
- Edit and delete controls

Users can add multiple slots. This supports several MapleStory reminders such as farming intervals, buffs, or cooldown checks.

## Hotkey Behavior

Each slot has a user-defined hotkey. When the hotkey is pressed:

- If the slot is idle, its timer starts.
- If the slot is already running, the current timer is reset to the full configured duration.
- If the slot just expired, pressing the hotkey starts it again.

Hotkeys are registered through Electron global shortcuts. The app must show whether each hotkey registration succeeded or failed. A hotkey can fail if Windows, MapleStory, or another app has already claimed it.

## Timer And Alarm Behavior

Each slot supports both presets and custom input:

- Presets: common quick durations such as 2, 5, and 10 minutes.
- Custom duration: minutes and seconds.

When a timer expires, the app triggers:

- A Windows desktop notification
- An audible alarm sound
- A visible expired state in the app UI

The app continues running when the window is minimized or closed to the tray. The tray menu includes:

- Open app
- Pause all timers
- Exit

Closing the window hides it to the tray. Explicit Exit from the tray menu fully quits the app.

## Persistence

Slot settings are saved locally so they survive app restarts:

- Slot name
- Hotkey
- Duration
- Enabled state

In-progress timers are not restored in the first version. The first version focuses on reliable background operation while the app process is running.

## Download And Distribution

The project should produce simple Windows download artifacts:

- A Windows installer for normal users
- A portable `.zip` or unpacked executable build for users who prefer not to install

The release output should be generated by a single package command. The final README should tell users which file to download and how to run it.

Code signing is out of scope for the first version, so Windows SmartScreen may warn on first launch.

## Architecture

Main process responsibilities:

- Create and manage the app window
- Keep the app alive in the tray
- Register and unregister global shortcuts
- Own timer state and expiration events
- Trigger notifications and alarm sounds
- Persist slot settings
- Expose a narrow IPC API to the renderer

Renderer responsibilities:

- Render the slot list and controls
- Capture hotkey input while editing a slot
- Send slot changes to the main process
- Display timer state pushed from the main process
- Show registration failures clearly

Timer state belongs in the main process so background behavior does not depend on renderer focus.

## Error Handling

The app should handle:

- Hotkey registration failure
- Duplicate hotkey assignment
- Invalid or zero duration
- Notification permission or OS notification failure
- Missing/corrupt settings file

Invalid settings should not crash the app. The app should fall back to an empty slot list or skip invalid slots and continue running.

## Testing Strategy

Automated tests cover:

- Hotkey string normalization
- Duplicate hotkey detection
- Slot persistence serialization and recovery
- Timer reset behavior when the same hotkey is pressed again
- Timer expiration event behavior

Manual verification covers:

- Electron window launches
- App hides to tray instead of quitting
- Tray menu opens and exits the app
- Global hotkeys work while another window is focused
- Notification and sound fire on expiration
- Windows build artifacts are produced

## Non-Goals For Version 1

- Cloud sync
- Mobile support
- Browser-only/PWA-only mode
- Restoring active countdowns after a full app restart
- Code signing or auto-update
